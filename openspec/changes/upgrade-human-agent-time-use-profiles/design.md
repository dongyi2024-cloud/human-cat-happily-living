## Context

`HumanAgent` currently uses `HUMAN_CONFIG["activity_patterns"]` to choose zones by coarse simulation phase. That model is useful for demos but not defensible for academic analysis because activity duration, outside time, work time, sleep time, and leisure time are not tied to a source dataset.

The new PRD in `2.0human_pro/PRD.md` changes the human model from time-of-day heuristics into a source-backed parameter pipeline. The implementation must preserve the existing standalone simulation workflow, keep tick records as the source of truth, support headless visualization, and keep cat behavior changes out of scope.

## Goals / Non-Goals

**Goals:**

- Add a `TimeUseParameterBuilder` that reads local time-use CSV files and profile mapping config.
- Support at least three data-supported human profiles: `remote_worker`, `service_worker`, and `default_china`.
- Convert raw source fields into activity budgets, zone budgets, and an activity schedule using explicit formulas.
- Store source metadata, formulas, normalization status, and model assumptions in output JSON.
- Update `HumanAgent` to follow an activity schedule and support `outside`.
- Expand tick records with human zone, activity, state, and profile ID while retaining existing downstream fields.
- Keep activity durations data-driven and residential zone mapping explicitly model-defined.
- Keep current cat profile behavior unchanged.

**Non-Goals:**

- No multi-person household model.
- No individual-level occupational prediction.
- No pet-care time synthesis unless a separate source dataset is provided.
- No commuting distance, door/path-to-door, or city transport model.
- No web scraping at runtime; source data must be local and versioned.
- No mixing US ATUS and China time-use data within one generated profile unless explicitly encoded as a documented model assumption.

## Decisions

### Use local versioned source CSV files

The builder will read from `data/` rather than fetching data from websites at runtime. This keeps profile generation reproducible and avoids silent changes when source websites update.

Alternative considered: download ATUS or government tables during simulation runs. That would make results less reproducible and introduce network fragility.

### Keep profile mapping separate from derived parameters

`config/human_profile_mapping.json` will bind profile IDs to dataset names, source tables, raw file names, occupation categories, and profile status. It will not store final activity budgets. Budgets are generated by `TimeUseParameterBuilder` from CSV source fields.

Alternative considered: store complete final profile JSON in config. That is easier, but it weakens traceability because final parameters can drift from source data without a formula trail.

### Treat activity-to-zone mapping as model assumption

`ACTIVITY_TO_ZONE_MAP` will map activities such as `sleep`, `home_work`, `household`, and `leisure` to residential zones. This mapping will be exported separately as `activity_to_zone_mapping_used.json` and referenced in `source_metadata.json`.

Alternative considered: derive zone use directly from time-use data. The PRD correctly notes that time-use surveys do not provide room-level residential behavior, so that would create false precision.

### Add logical outside state before physical door modeling

When activity is `outside`, the human agent will emit `human_behavior = "外出"`, `human_zone = "outside"`, and `human_state = "outside"`. Indoor heatmap and indoor movement will not update while outside. `human_x` and `human_y` in CSV will be empty/null-compatible for outside ticks.

Alternative considered: move the agent to a fixed door or edge coordinate. The current floor plan has no door semantic, so a logical state is more honest and less likely to pollute indoor metrics.

### Generate schedule from data-driven durations with model-defined order

The first implementation will generate an activity schedule using a fixed order template such as sleep → household/care → work/outside/education → household/care → leisure → sleep/rest. Durations are data-driven; order is model-defined and exported as an assumption.

Alternative considered: infer exact time-of-day distributions. That requires more detailed source tables and is out of scope for the current change.

### Preserve analyzer compatibility

Existing tick fields `tick`, `human_x`, `human_y`, and `human_behavior` will remain present. Downstream analyzers must tolerate outside ticks by skipping or otherwise safely handling null human coordinates. New human behavior labels will be added to behavior weights where needed.

Alternative considered: only write a separate human timeline file. That would keep old CSV unchanged but fragment the tick-level source of truth.

## Risks / Trade-offs

- Missing or incomplete source CSV files → fail with a clear error listing required files and profile mapping entries, except for explicitly documented seed fixtures used for tests.
- Source tables do not sum to 24 hours → record `normalization_applied` and the exact scale formula in metadata.
- Occupation profiles may be weakly supported by public data → mark `profile_status` as `model_assumption` and exclude those profiles from default paper-ready comparisons.
- Outside ticks may break old analyzers that assume numeric coordinates → update loading logic and tests to handle outside ticks without inventing indoor positions.
- Activity order can bias spatial sequences → export the order template as a model assumption and keep later refinement possible.
- Running all outputs in `result/` vs `outputs/` can confuse workflows → keep simulation output configuration explicit and respect the project rule that test artifacts go to `result/`.

## Migration Plan

1. Add local data/config scaffolding for the supported profiles.
2. Implement the builder and verify generated activity budgets sum to `total_ticks`.
3. Update `HumanAgent` to accept generated profiles while preserving default construction.
4. Expand tick records and output files.
5. Update analyzer handling and behavior weights for new human labels.
6. Run comparison tests for `remote_worker`, `service_worker`, and `default_china`.

Rollback is straightforward: keep the current `HumanAgent` path as fallback until the generated profile path is verified, then remove fallback only if the specs require it.
